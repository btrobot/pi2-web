# 导航与模式切换文案收敛建议

> 适用范围：`app/i18n_registry.py`、`api/templates/index.html`
> 目标：分析当前“完整中文描述”是否过长，并给出更适合导航栏与模式切换提示的命名方案

## 1. 结论

**是的，改成类似 `文字→语音`、`语音→文字` 这类更短的结构，整体会更好。**

原因不是“更短就一定更高级”，而是当前页面里的这批文案承担的是：
- 快速扫读
- 快速区分
- 快速做决定

而不是承担完整说明。

对于导航栏和模式切换区，这类位置最需要的是：
- **低阅读成本**
- **高可扫读性**
- **同构命名**
- **一眼看懂输入与输出关系**

当前“完整中文描述”更像说明文，不像导航标签。

---

## 2. 当前文案存在的问题

当前导航主标签包括：
- 同语言文字转语音
- 同语言语音转文字
- 异语言文字转语音
- 异语言语音转文字
- 异语言文本互译
- 异语言语音互译

当前模式 leaf 标签包括：
- 中文文字转中文语音
- 英文文字转英文语音
- 中文语音转英文语音
- 英文语音转中文文字
- ……

这些文案的问题主要有 4 个。

### 2.1 太像“句子”，不像“标签”

导航标签本质上应该是“瞬时识别单元”，而不是需要完整读完的短句。

比如：
- `同语言文字转语音`

用户必须顺序处理：
1. 同语言
2. 文字
3. 转
4. 语音

而 `文字→语音` 基本是瞬时识别。

### 2.2 视觉长度接近，扫读区分度不够

当前 6 个标签几乎都长得很像：
- 同语言 + X + 转 + Y
- 异语言 + X + 转 + Y

用户在扫左侧导航时，更容易感受到“它们都差不多长、差不多像”，而不是快速抓到关键差异。

### 2.3 关键信息顺序不符合第一眼识别

用户在导航上最先想确认的通常是：
- 输入是什么
- 输出是什么

但现在标签把“同/异语言”放在最前面，把真正决定任务类型的“文字/语音”放在后面。

这会导致第一眼抓不到任务本体。

### 2.4 模式切换区重复啰嗦，认知负担高

`api/templates/index.html` 里当前 mode picker 会直接渲染完整 leaf 名称；summary 还会显示：
- group 标签 / leaf 标签

也就是用户会多次看到类似：
- 异语言语音转文字 / 中文语音转英文文字

这类重复说明在“第一次介绍”时有价值，但在“切换模式”这种高频操作里太重了。

---

## 3. 为什么“文字→语音”这类写法更合适

### 3.1 箭头结构天然表达转换关系

`文字→语音` 的优势在于它几乎不需要语言解释，用户一眼就能理解：
- 左边是输入
- 右边是输出

这比“文字转语音”更像界面语言，而不是叙述语言。

### 3.2 更适合做一组并列项

如果一组标签都采用同一结构：
- 文字→语音
- 语音→文字
- 文字→文字
- 语音→语音

用户的对比成本会明显下降。

### 3.3 更适合做多层信息拆分

短标签适合做主标签；语言方向、是否跨语种、具体语义，可以放在次级信息层里。

例如：
- 主标签：`文字→语音`
- 次标签：`中文→英文`

这样比把所有信息压进一个长句里更清晰。

---

## 4. 但不能只把长文案机械替换成短文案

这里有一个关键点：

**如果只把“同语言文字转语音”改成“文字→语音”，但页面结构完全不变，仍然会有信息歧义。**

因为当前左侧有 6 个分组，其中至少会出现：
- 同语言文字转语音
- 异语言文字转语音

如果都缩成 `文字→语音`，那两项就撞名了。

所以正确做法不是“纯缩写”，而是：
- **主识别标签简化**
- **差异信息下沉到副标题 / tag / pill / 辅助说明**

也就是说，应该把文案结构从“一行塞满所有信息”改成“两层表达”。

---

## 5. 导航栏建议

## 5.1 如果暂时不改页面结构，只改文案

如果现在还保留 6 组导航，建议不要直接改成完全同名的 4 个箭头标签。

建议使用：

### 方案 A：短主标签 + 语言范围前缀
- 同语 `文字→语音`
- 同语 `语音→文字`
- 异语 `文字→语音`
- 异语 `语音→文字`
- 异语 `文字→文字`
- 异语 `语音→语音`

优点：
- 比当前短很多
- 保留同/异语种差异
- 仍然和现有 group 结构兼容

缺点：
- 还是有一点“系统分组味”
- `同语 / 异语` 仍然不是用户最自然的第一识别项

### 方案 B：主标签短写，副标签解释
主标签：
- `文字→语音`
- `语音→文字`
- `文字→语音`
- `语音→文字`
- `文字→文字`
- `语音→语音`

副标签：
- 同语种
- 同语种
- 跨语种
- 跨语种
- 跨语种
- 跨语种

优点：
- 主信息最短
- 扫读效率最高
- 次信息不抢视线

缺点：
- 需要导航视觉样式支持两行或 subtitle

**如果只做低风险改文案，我更推荐方案 B。**

---

## 5.2 如果允许顺便优化导航结构

更推荐直接把导航改成 4 个任务类型：
- `文字→语音`
- `语音→文字`
- `文字→文字`
- `语音→语音`

然后把：
- 中文→中文
- 中文→英文
- 英文→中文
- 英文→英文

放进主内容区做模式切换。

这是更符合用户心智的结构，因为：
- 导航负责选任务类型
- 主区负责选语言方向

职责更清楚。

---

## 6. 模式切换区建议

当前模式切换区最大的问题不是“长”，而是：
- 太长
- 太重复
- 一次展示了 group 和 leaf 两层近似信息

例如：
- Summary：异语言语音转文字 / 中文语音转英文文字
- Mode pill：中文语音转英文文字

对用户来说，这相当于被反复告知同一件事。

### 6.1 模式切换最适合显示什么

模式切换区应该优先回答：
- 当前是哪种语言方向
- 切过去后会得到什么

所以建议把 leaf mode 改成：
- `中→中`
- `中→英`
- `英→中`
- `英→英`

前提是当前导航已经确定了任务类型，比如当前在：
- `文字→语音`

这时 mode pill 只需要显示语言方向。

这是最省认知的组合：
- 导航：任务类型
- mode pill：语言方向

两者相乘即可唯一确定模式。

### 6.2 如果暂时不改结构，mode pill 也应缩短

如果当前页面仍保留 group + leaf 结构，mode pill 也建议改成短格式：
- `中→中`
- `英→英`
- `中→英`
- `英→中`

然后在当前标题 / summary 里补一句完整说明，例如：
- 当前任务：文字→语音
- 当前方向：中文→英文

而不是让按钮本身承担整句说明。

---

## 7. Summary / 模式切换提示文案建议

当前 summary caption 类似：
- `异语言语音转文字 / 中文语音转英文文字`

这个结构的问题是：
- 两段都在说“这是一个什么模式”
- 信息增量很小
- 文案像系统拼装，不像界面提示

建议改成下面这种结构。

### 7.1 推荐写法：任务 + 方向 + 当前输入状态

例如：
- 当前任务：`语音→文字`
- 语言方向：`中文→英文`
- 当前输入：`已选择录音 #3`

或者合成一句：
- `语音→文字 · 中文→英文`

如果有输入状态，再补第二行：
- `已选择录音 #3，可开始转换`

### 7.2 不建议继续使用“同语言 / 异语言”作为主提示

原因是用户在已经选好模式后，并不关心分类学描述；他更关心：
- 我这次是不是中文变英文
- 最终出来的是文字还是语音

所以：
- `异语言语音转文字`

不如：
- `语音→文字 · 中文→英文`

后者更直接、更接近操作结果。

---

## 8. 推荐命名体系

建议建立统一命名规则：

### 8.1 一级导航：任务类型
统一用：
- `文字→语音`
- `语音→文字`
- `文字→文字`
- `语音→语音`

### 8.2 二级切换：语言方向
统一用：
- `中→中`
- `中→英`
- `英→中`
- `英→英`

### 8.3 页面摘要 / 小标题：组合表达
统一用：
- `文字→语音 · 中文→英文`
- `语音→文字 · 英文→中文`

### 8.4 状态提示：动作表达
统一用：
- `等待输入文本`
- `正在录音 12 秒`
- `已选择录音 #3，可开始转换`
- `正在生成英文语音`
- `已完成，可播放或下载`

这四层一旦统一，整页会更像一个产品，而不是一堆接口标签拼起来的壳。

## 9. 文案替换建议表

| 场景 | 当前写法 | 建议写法 | 说明 |
|---|---|---|---|
| 导航组 | 同语言文字转语音 | 文字→语音 + 副标签“同语种” | 主标签缩短，差异下沉 |
| 导航组 | 异语言语音转文字 | 语音→文字 + 副标签“跨语种” | 主体先行 |
| 导航组 | 异语言文本互译 | 文字→文字 + 副标签“跨语种” | 去掉“互译”术语感 |
| 导航组 | 异语言语音互译 | 语音→语音 + 副标签“跨语种” | 更直观 |
| mode pill | 中文文字转英文语音 | 中→英 | 语言方向即可 |
| mode pill | 英文语音转中文文字 | 英→中 | 同上 |
| summary | 异语言语音转文字 / 中文语音转英文文字 | 语音→文字 · 中文→英文 | 信息去重 |
| 状态提示 | 使用录音 #3 / 模式切换 | 已选择录音 #3，请选择语言方向 | 从系统提示改成任务提示 |
| 区块标题 | 模式切换 | 语言方向 | 如果当前导航已确定任务类型 |

## 10. 对现有实现的具体建议

结合当前代码，建议优先改这几类文案层：

### 10.1 `nav.*`
当前在 `app/i18n_registry.py` 中：
- `nav.same_text_to_speech`
- `nav.same_speech_to_text`
- `nav.cross_text_to_speech`
- `nav.cross_speech_to_text`
- `nav.cross_text_to_text`
- `nav.cross_speech_to_speech`

建议不要继续使用完整句式。

### 10.2 `mode.*`
当前：
- `mode.tts_zh_zh = 中文文字转中文语音`
- `mode.mt_tts_zh_en = 中文文字转英文语音`
- ……

如果 mode pill 继续直接渲染 `mode.*`，会一直显得很长。

建议拆成两套：
- 完整说明：用于无障碍 / debug / title / history
- 短标签：用于 mode picker 按钮

### 10.3 summary caption
当前 `renderSummary()` 是把 group label 和 leaf label拼在一起。

建议改成：
- 任务类型短标签
- 语言方向短标签

不要继续拼完整句子。

### 10.4 “使用录音 / 模式切换”提示
当前提示类似：
- `使用录音 #3 / 模式切换`

这句话的产品语言比较弱。

建议改成：
- `已选择录音 #3，请选择语言方向`
- 或 `已选择录音 #3，可切换为中→英 / 英→中`

用户更容易理解下一步该做什么。

---

## 11. 最终建议

如果只做一个判断：

### 判断 1：导航栏用完整中文描述是否啰嗦？
**是，偏啰嗦。**

### 判断 2：改成类似 `文字→语音` 会不会更好？
**会更好，但要配合信息分层，不能机械替换。**

### 判断 3：模式切换提示是不是也有同样问题？
**是，同样存在“过长、重复、系统味太重”的问题。**

### 建议拍板方案
- 导航主标签：改成 `文字→语音 / 语音→文字 / 文字→文字 / 语音→语音`
- 同语种 / 跨语种：下沉为副标签或直接移入主区
- mode pill：改成 `中→中 / 中→英 / 英→中 / 英→英`
- summary：改成 `任务类型 · 语言方向`
- 状态提示：改成“下一步动作提示”，不要再用系统术语

---

## 12. 一句话标准

**凡是用户需要频繁扫读和切换的位置，用短结构；凡是用户需要确认细节的位置，再给完整说明。**

导航和模式切换显然属于前者，所以应该收短，不应该继续写成完整中文说明句。
