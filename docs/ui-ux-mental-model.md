# 页面应该如何收口到更符合用户心智

> 适用范围：当前 `api/templates/index.html` 单页 Web UI  
> 目标：在不推翻现有后端 mode / API 的前提下，把页面从“技术模式导航壳”收口成“用户任务工作台”

## 1. 一句话结论

当前页面更像“把内部 pipeline 直接暴露给用户”，而不是“让用户完成一次中英双语处理任务”。

更符合用户心智的收口方向应该是：**前端只暴露任务意图、输入方式、输出结果，隐藏 mode_key / pipeline / same vs cross 等内部结构**，把页面收口为一个“离线双语转换工作台”。

---

## 2. 当前页面的核心问题

### 2.1 导航按系统实现分组，不按用户任务分组

当前左侧一级导航是：
- 同语种文本转语音
- 同语种语音转文本
- 异语种文本转语音
- 异语种语音转文本
- 异语种文本互译
- 异语种语音互译
- 录音
- 历史

这套分类是**研发视角**，不是**用户视角**。

用户的想法通常不是：
- “我要进入 cross_speech_to_text”

而是：
- “我想把这段中文读出来”
- “我想把这段英文翻成中文”
- “我说一句中文，给我英文文字/英文语音”

也就是说，用户心智是“我要完成什么任务”，不是“我要选择哪条 pipeline”。

### 2.2 选择链路太长，决策被拆碎

当前用户往往要经历：
1. 先选 group
2. 再选 leaf mode
3. 再决定输入方式
4. 再看输出区里到底会出现文本还是音频

这会造成一个问题：**用户在提交前，很难快速确认“这次到底会产出什么”**。

### 2.3 页面把“主任务”和“资源管理”放在同一层级

“录音”“历史”目前与功能模式并列在一级导航里，但它们本质上不是主任务，而是：
- 录音：输入资产 / 素材库
- 历史：结果资产 / 可追溯记录

这两者更像**辅助资源区**，不应该和“翻译 / 识别 / 朗读”并列抢心智入口。

### 2.4 暴露了太多内部术语

当前页面里出现了大量内部结构感较强的信息，例如：
- `mode_key`
- 同语种 / 异语种
- 输入类型 / 输出类型 pill
- summary 中重复展示 group + leaf

这些信息对开发和调试有用，但对用户来说是噪音。

### 2.5 结果区不是“任务结果”，而是“字段容器”

当前输出区会分别展示：
- source text
- output text
- source audio
- output audio

这对系统结构很友好，但对用户来说，真正关心的顺序通常是：
1. 我输入了什么
2. 系统识别/翻译成了什么
3. 我现在可以复制、播放、下载什么

也就是说，用户更需要**结果叙事**，不是**字段罗列**。

### 2.6 状态反馈偏技术，不够任务化

当前状态更多是 `Ready / Processing / Error`。

但用户真正需要的是：
- 等待你输入文本
- 正在录音（12 秒 / 180 秒）
- 正在把中文翻成英文语音
- 已生成英文结果，可播放或复制
- 历史已保存

**状态应该回答“系统现在在帮我做什么”**，而不是只回答“系统当前健康否”。

---

## 3. 用户更自然的心智模型

对于这个产品，用户的自然思考顺序通常是：

1. **我要处理什么内容？**
   - 文本
   - 语音

2. **我要得到什么结果？**
   - 文本
   - 语音

3. **涉及哪两个语言？**
   - 中文 -> 中文
   - 英文 -> 英文
   - 中文 -> 英文
   - 英文 -> 中文

4. **我怎么拿到结果？**
   - 直接看
   - 复制
   - 播放
   - 下载
   - 留到历史里

所以页面的理想结构应该围绕这 4 个问题展开，而不是围绕 ASR / MT / TTS 的组合关系展开。

---

## 4. 推荐的收口策略

## 4.1 页面只保留一个主工作台

建议把页面收口为一个主工作台，名称可以直接叫：
- **离线中英转换台**
- **语音 / 文本转换工作台**
- **双语语音处理台**

用户进入页面后，不应该先面对“导航树”，而应该先面对一个明确任务入口。

## 4.2 用“任务卡片 / 顶部标签”替代左侧 6 类技术分组

建议一级任务收口为 4 类：

1. **文本朗读**
   - 文本 -> 语音
   - 覆盖：中文朗读、英文朗读、中文转英文语音、英文转中文语音

2. **语音识别**
   - 语音 -> 文本
   - 覆盖：中文识别、英文识别、中文语音转英文文本、英文语音转中文文本

3. **文本翻译**
   - 文本 -> 文本 / 语音
   - 如果要进一步收口，也可以把“翻译后输出为文本/语音”做成一个结果类型开关

4. **语音翻译**
   - 语音 -> 文本 / 语音
   - 同样用结果类型开关控制最终产物

如果希望收得更紧，还可以只保留 **2 个一级入口**：
- 文本输入
- 语音输入

然后在主区里通过：
- 输出类型（文本 / 语音）
- 语言方向（中 -> 中 / 英 -> 英 / 中 -> 英 / 英 -> 中）

来推导最终 mode。

**推荐优先采用 2 个一级入口方案。**
因为它最符合“我现在手里拿着什么输入”的第一判断。

---

## 5. 推荐信息架构

## 5.1 推荐结构（最佳收口版）

### 一级：输入入口
- 文本输入
- 语音输入

### 二级：任务参数
- 来源语言
- 目标语言
- 输出类型（文本 / 语音）

### 三级：辅助资源
- 录音库
- 历史记录
- 设置 / 帮助

### 系统内部：继续保留现有 frozen mode_key
前端根据用户选择自动映射：
- 文本输入 + 中文 -> 中文 + 语音输出 => `tts_zh_zh`
- 文本输入 + 中文 -> 英文 + 文本输出 => `mt_zh_en`
- 文本输入 + 中文 -> 英文 + 语音输出 => `mt_tts_zh_en`
- 语音输入 + 中文 -> 中文 + 文本输出 => `asr_zh_zh`
- 语音输入 + 中文 -> 英文 + 文本输出 => `asr_mt_zh_en`
- 语音输入 + 中文 -> 英文 + 语音输出 => `asr_mt_tts_zh_en`

这样可以做到：
- **后端 contract 不变**
- **前端心智大幅简化**

## 6. 推荐页面骨架

```text
┌────────────────────────────────────────────┐
│ 离线中英转换台                    中文 / EN │
├────────────────────────────────────────────┤
│ [文本输入] [语音输入]                        │
│                                            │
│ Step 1 选择语言方向                         │
│ [中文→中文] [中文→英文] [英文→中文] [英文→英文] │
│                                            │
│ Step 2 选择结果形式                         │
│ [输出文本] [输出语音]                       │
│                                            │
│ Step 3 提供输入                             │
│ - 文本：输入 / 上传 txt                     │
│ - 语音：录音 / 上传 wav / 从录音库选择       │
│                                            │
│ [开始转换] [清空]                           │
├────────────────────────────────────────────┤
│ 本次结果                                     │
│ - 识别文本 / 翻译文本 / 音频播放器            │
│ [复制文本] [下载音频] [保存到历史]            │
├────────────────────────────────────────────┤
│ 最近记录（3 条）   [查看全部]                │
└────────────────────────────────────────────┘
```

这个结构的关键点是：
- 页面主轴只有一条：**配置任务 -> 提交输入 -> 查看结果**
- 历史和录音不再抢一级心智
- 用户在提交前能非常明确地知道“这次会输出文本还是语音”

## 7. 具体 UI/UX 建议

### 7.1 隐藏 mode_key，不再面向用户展示

当前 `active-mode-key` 这种展示对用户没有价值。

建议：
- 对外隐藏 mode_key
- 仅在开发环境或 debug 面板里保留

### 7.2 把 summary 区从“状态卡”改成“本次任务配置卡”

当前 summary 更像系统摘要。

建议改成：
- 当前任务：语音输入 -> 英文文本
- 语言方向：中文 -> 英文
- 当前输入来源：浏览器录音 / 上传文件 / 已保存录音 #3
- 当前将生成：英文文本

这样 summary 才是帮助用户确认动作，而不是展示系统元信息。

### 7.3 把“帮助 / 设置”降级到次级入口

当前头部 3 个按钮里，帮助、设置、语言切换都比较显眼。

建议：
- 保留语言切换
- 帮助 / 设置合并到一个“更多”入口，或右上角抽屉
- 默认不抢主流程注意力

### 7.4 录音库不要独立成主导航，应该嵌入语音输入流程

更符合用户心智的方式是：

在“语音输入”区里提供三个并列来源：
- 立即录音
- 上传 WAV
- 从录音库选择

而不是先去“录音”页面，再回到转换页面。

### 7.5 历史记录应该是结果回看，不是主流程入口

建议：
- 首页仅保留“最近记录 3 条”
- “查看全部历史”进入独立页面或抽屉
- 历史项按“任务结果卡片”展示，而不是按技术字段展示

每条历史优先显示：
- 做了什么（如：中文语音 -> 英文语音）
- 什么时候做的
- 核心结果预览
- 可执行动作（播放 / 复制 / 下载 / 删除）

### 7.6 按钮文案改为动作导向

建议替换：
- `Start conversion` -> `开始生成` / `开始转换`
- `Save input` ->
  - 文本模式：去掉，默认随任务保存即可
  - 语音模式：改为 `保存到录音库`
- `Reset` -> `清空本次内容`
- `View all history` -> `查看全部记录`

重点：**一个按钮只表达一个稳定动作**，不要在不同模式下让按钮语义变化过大。

### 7.7 输出区要按“结果优先级”排列

建议输出顺序固定为：
1. 最终结果
2. 可操作动作
3. 中间结果（如果有）
4. 原始输入预览

例如：
- 如果任务是中文语音 -> 英文语音
  - 第一屏先给英文音频播放器
  - 第二给英文文本
  - 第三给识别出的中文文本
  - 最后再给原始输入音频

不要让“原始输入”和“最终结果”在视觉上同权。

### 7.8 状态提示改为任务语言

建议替换为：
- 等待输入文本
- 正在录音（18 秒）
- 正在识别中文语音
- 正在翻译成英文
- 正在生成英文语音
- 已完成，可播放或下载
- 保存成功，已进入历史记录

这会明显提升可理解性。

## 8. 对现有代码结构的落地建议

这个收口不需要先改后端，可以先做 **纯前端收口**。

## 8.1 可以保留的部分
- `mode_registry.py` 的 12 个 frozen mode
- `/api/conversions/text`
- `/api/conversions/speech`
- `/api/recordings`
- `/api/history`
- 现有历史 / 录音数据结构

## 8.2 建议前端新增一层“用户任务配置 -> mode_key”映射

前端内部维护：

```text
input_kind + source_lang + target_lang + output_kind -> mode_key
```

这样可以：
- 不再把 group_key 暴露给 UI
- 不再要求用户先选 group 再选 leaf
- 把当前 12 个模式当作实现细节

## 8.3 建议删除或降级的现有 UI 元素
- 左侧 6 个 group 导航
- `active-mode-key` 展示
- summary 中的内部 meta pill
- 与主流程并列的“录音 / 历史”一级导航
- 头部独立 help / settings 强入口

## 8.4 建议保留但改位置的能力
- 最近历史：保留在主工作台底部
- 全部历史：独立管理页 / 抽屉
- 录音复用：保留在语音输入区
- 语言切换：保留在头部

---

## 9. 推荐改版优先级

### P0：先做“心智收口”，不改后端
1. 隐藏 mode_key / group_key
2. 把 6 类模式入口收成 2 类：文本输入 / 语音输入
3. 增加语言方向和输出类型选择器
4. 录音库嵌入语音输入区
5. 历史降级为最近记录 + 查看全部

### P1：再做“结果叙事化”
1. 重排输出区层级
2. 改写状态文案
3. 统一按钮文案
4. 减少 summary 冗余信息

### P2：最后做“体验打磨”
1. 录音中实时计时与上限提示更明显
2. 转换中的阶段提示（识别 / 翻译 / 合成）
3. 历史筛选（按文本/语音、按语言方向）
4. 空状态插图 / 示例输入 / 快捷示例

---

## 10. 最终建议（建议拍板版本）

如果只给一个明确建议，我会建议：

**把当前页面从“模式导航页”改成“单工作台页”。**

核心交互改为：
- 先选输入类型（文本 / 语音）
- 再选语言方向（中->中 / 中->英 / 英->中 / 英->英）
- 再选输出类型（文本 / 语音）
- 系统自动决定 mode_key
- 录音和历史作为辅助资源嵌入，不再抢一级入口

这会让产品从“开发者能理解的系统壳”收口成“用户一眼就知道怎么用的任务工具”。

---

## 11. 附：一句话判断标准

如果一个第一次打开页面的用户，不需要理解：
- ASR / MT / TTS
- same / cross
- mode_key
- group / leaf

也能在 5 秒内知道“我该点哪里开始”，那么这次收口就是成功的。
