# 右侧主工作台低保真线框图建议

> 适用页面：`api/templates/index.html` 当前主工作台右侧区域  
> 目标：减少重复说明、压缩无效留白、把“输入-操作-结果”收拢到首屏主路径中

---

## 1. 设计结论

建议把当前右侧区域从：
- 面包屑
- 解释摘要卡
- 输入卡
- 控制卡
- 输出卡
- 最近历史卡

收口为：
- **任务头部**
- **左右双栏主工作台**
  - 左：输入 + 操作
  - 右：结果
- **最近历史**

这样能解决两个核心问题：
1. 顶部重复说明
2. 控制区占位过大，导致输出下沉

---

## 2. 信息架构建议

### 一级结构
1. 任务头部
2. 主工作台
3. 最近历史

### 主工作台内部结构
- 左栏：输入与操作
- 右栏：输出结果

### 辅助内容
- 帮助 / 设置：保留为可展开区，不占主工作台首屏
- 全量历史：独立视图

## 4. 移动端低保真线框图

移动端不建议双栏，应该改成单列顺序：

```text
┌──────────────────────────────┐
│ 语音→文字 · 中→英   [就绪]    │
│ 已选择录音 #3                │
├──────────────────────────────┤
│ 输入与操作                    │
│ - 语言方向                    │
│ - 输入区                      │
│ - 按钮条                      │
│ - 状态提示                    │
├──────────────────────────────┤
│ 输出结果                      │
│ - 最终结果                    │
│ - 中间结果                    │
│ - 原始输入预览                │
├──────────────────────────────┤
│ 最近历史                      │
│ [查看全部]                    │
└──────────────────────────────┘
```

### 移动端原则
- 输入与操作必须连续
- 输出紧跟其后
- 历史放最底部
- 不要插入一个“独立控制卡”打断流程

---

## 5. 模块职责建议

## 5.1 顶部任务头部

### 建议保留
- 当前任务类型：`文字→语音 / 语音→文字 / 文字→文字 / 语音→语音`
- 当前语言方向：`中→英 / 英→中 / 中→中 / 英→英`
- 当前状态：`就绪 / 处理中 / 异常`
- 可选一行动态提示：`已选择录音 #3，可开始转换`

### 建议去掉
- 主工作台里的面包屑
- 单独的 explanation / summary 卡重复描述
- 冗余 meta pill

### 原因
主工作台是“工具首页”，不是深层信息层级页，不需要两层“你在这里”的说明。

---

## 5.2 输入与操作区

### 应包含
- 语言方向切换
- 文本/语音输入控件
- 录音库选择
- 主按钮区
- 状态提示

### 不应再拆出去的内容
- 开始转换
- 重置
- 保存录音
- 当前状态提示

### 原因
这些都属于同一个动作闭环：
**配置输入 → 触发动作 → 等待状态**

---

## 5.3 输出区

### 推荐展示顺序
1. 最终结果
2. 结果操作
3. 中间结果
4. 原始输入预览

### 示例
#### 语音→文字
- 第一层：翻译/识别结果文本
- 第二层：复制按钮
- 第三层：原始输入音频（可选）

#### 语音→语音
- 第一层：目标音频播放器
- 第二层：目标文本
- 第三层：识别文本
- 第四层：原始输入音频

### 原因
输出区必须围绕“用户最想得到的结果”组织，而不是围绕系统字段组织。

---

## 5.4 最近历史区

### 建议位置
- 主工作台下方整行

### 建议内容
- 最近 3 条即可
- 每条显示：任务类型 + 方向 + 时间 + 结果预览
- 右侧给 `查看全部`

### 不建议
- 和输入/输出区并列放在首屏主网格里

### 原因
历史是辅助回看，不是当前主任务。

---

## 6. 布局比例建议

## 6.1 桌面端
推荐：
- 左栏输入与操作：**42%**
- 右栏输出结果：**58%**

原因：
- 这个产品的价值更偏“结果消费”
- 输出文本、播放器、下载等通常更需要空间

## 6.2 卡片高度策略
当前问题之一是统一 `min-height` 让小内容卡显得过高。

建议：
- 输入卡：自适应高度
- 输出卡：自适应高度，但优先占视觉重点
- 历史卡：内容驱动高度
- 不再让按钮卡拥有独立大高度

---

## 7. 视觉权重建议

### 一级重点
- 输入与操作
- 输出结果

### 二级重点
- 顶部任务头部
- 最近历史

### 三级重点
- 帮助
- 设置
- 约束说明

### 原则
真正的大卡片最好只保留两块：
- 输入
- 输出

否则页面会显得“块很多，但重点不明确”。

---

## 8. 与现有页面的映射关系

## 当前结构
- breadcrumb
- summary-card
- input-panel
- control-panel
- output-panel
- history-panel

## 建议目标结构
- task-header（由 breadcrumb + summary 合并）
- input-panel（吸收 control-panel）
- output-panel
- history-panel

## 直接对应关系
- `app-breadcrumb`：主工作台隐藏或仅在历史页保留
- `app-summary`：改造成 task-header
- `app-control-panel`：删除，并入 `app-input-panel`
- `app-output-panel`：提升为首屏主视觉区域
- `app-history-panel`：保持，但移动到主工作台底部整行

---

## 9. 改造优先级建议

### P0（马上该做）
1. 去掉主工作台面包屑
2. summary 改成紧凑任务头部
3. 控制卡并入输入卡
4. 输出区上移到首屏主视野

### P1（下一步优化）
1. 输出区按“最终结果优先”重排
2. 最近历史改成底部整行
3. 帮助/设置改成抽屉或折叠面板

### P2（体验精修）
1. 状态信息更细化
2. 录音过程视觉强调
3. 空状态优化
4. 更强的响应式布局

---

## 10. 推荐拍板方案

如果只选一个方案，我建议：

### 采用“紧凑任务头部 + 左右双栏工作台 + 底部最近历史”

这是目前最符合你这个产品形态的布局：
- 工具型
- 单任务流
- 输入和输出强关联
- 结果是核心价值

---

## 11. 一句话标准

**用户在首屏应该同时看到：现在在做什么、从哪里输入、点哪里开始、结果会出现在哪里。**

如果做不到这一点，布局就还没有真正服务任务流。
