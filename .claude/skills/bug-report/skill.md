# Bug Report Skill

Bug 报告工作流，记录和跟踪缺陷。

## 触发方式

```
/bug-report
/bug-report 账号创建后列表不刷新
/bug-report E2E 测试登录失败
```

## Bug 严重性定义

| 等级 | 标识 | 定义 | 示例 |
|------|------|------|------|
| S1 | 🔴 关键 | 崩溃、数据丢失、阻塞 | 闪退、数据丢失、核心功能不可用 |
| S2 | 🟠 高 | 重大影响、功能损坏 | 登录失败、支付异常、数据显示错误 |
| S3 | 🟡 中 | 中等影响、体验问题 | UI 显示异常、性能问题、边界条件 |
| S4 | 🟢 低 | 轻微问题、建议 | 拼写错误、样式调整、用户体验优化 |

## Bug 报告模板

```markdown
## Bug Report: [标题]

**报告日期**: YYYY-MM-DD
**报告人**: [姓名]
**严重性**: S1/S2/S3/S4
**状态**: 待确认 / 进行中 / 已修复 / 已关闭

### 概述
[一行简短的描述]

### 复现步骤
1. [步骤 1]
2. [步骤 2]
3. [步骤 3]

### 预期行为
[期望发生什么]

### 实际行为
[实际发生了什么]

### 环境
- 操作系统: [OS]
- 浏览器: [Browser]
- 应用版本: [Version]

### 截图/日志
[附加相关截图或日志]

### 根因分析
[初步分析的可能原因]

### 修复建议
[如果有建议的修复方案]

### 相关链接
- Jira: [链接]
- PR: [链接]
```

## 执行步骤

### Step 1: 确认 Bug

验证 Bug 是否可复现：
- 检查环境是否正确
- 按照步骤复现
- 收集必要的日志

### Step 2: 分类 Bug

根据严重性定义分类：
- S1: 立即停止工作，通知负责人
- S2: 当天处理
- S3: 本周处理
- S4: 下个 Sprint 处理

### Step 3: 分配 Bug

根据功能模块分配：
| 模块 | 负责人 |
|------|--------|
| 前端页面 | Frontend Lead |
| API 端点 | Backend Lead |
| 自动化脚本 | Automation Developer |
| 安全问题 | Security Expert |

### Step 4: 跟踪修复

记录修复进度：
```markdown
## Bug 跟踪

| ID | 标题 | 严重性 | 状态 | 负责人 | 创建日期 |
|----|------|--------|------|--------|----------|
| BUG-001 | [标题] | S2 | 进行中 | Backend Lead | 2024-01-15 |
```

## Bug 关闭条件

| 状态 | 条件 |
|------|------|
| 已修复 | 代码已合并，测试通过 |
| 不复现 | 无法复现，关闭 |
| 重复 | 与已有 Bug 重复，关闭 |
| 设计如此 | 确认为设计行为，关闭 |

## 输出

更新 Bug 跟踪列表：

```markdown
## Bug 跟踪 - [日期]

### 本周新增
- BUG-XXX: [标题] (S2)

### 进行中
- BUG-XXX: [标题] (S2) - Backend Lead

### 已关闭
- BUG-XXX: [标题] - 已修复
```

同时更新 `production/session-state/active.md` Bug 记录。
